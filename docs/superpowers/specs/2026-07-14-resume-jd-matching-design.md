# 简历与 JD 精准匹配 Agent 设计说明

## 目标

在保留 `agent.py`、`tools.py`、`model_client.py` 多文件职责划分的前提下，将现有粗略模型打分改造成稳定、可解释的混合评分流程，并自动完成精准评分、STAR 项目经历改写、增补关键词生成和 `result.txt` 导出。

程序直接读取项目根目录下的 `resume.txt` 与 `jd.txt`，不再要求用户在界面粘贴文本或逐步触发功能。

## 总体方案

采用“规则评分 + 模型改写”的混合架构：

- 确定性规则负责 JD 分类、硬性门槛判定、技能同义词匹配、分项计分、扣分和总分封顶。
- 本地 Ollama 模型只补充复杂 JD 条款的结构化抽取，并依据简历已有事实生成自然的 STAR 项目描述。
- 模型给出的复杂条款必须包含 JD 原文证据；没有证据的模型结论不能触发硬性扣分。
- 模型不可用或返回非法内容时，规则评分、缺失项分析、关键词清单和文件导出仍可完成，STAR 改写使用保守模板降级。

## 文件职责

### `src/model_client.py`

- 封装 Ollama 兼容的 OpenAI 客户端。
- 从环境变量读取服务地址、API Key 和模型名。
- 提供普通文本调用和结构化 JSON 调用。
- 统一处理超时、空响应、Markdown 代码围栏和非法 JSON。
- 不包含任何评分权重或业务规则。

### `src/tools.py`

- 保存软件测试技能词库、同义词组、学历层级和专业族映射。
- 将 JD 拆为硬性门槛、硬性技术技能和软素质。
- 从简历提取学历、工作年限、专业、技能和项目上下文证据。
- 计算硬性条件、技术关键词和项目经历三个分项。
- 构建 STAR 改写提示；模型失败时生成不虚构事实的规则模板。
- 生成增补关键词清单和中文总结所需数据。

### `src/agent.py`

- 串行编排完整流程，不要求调用方逐步触发。
- 组装稳定的标准 JSON 结果。
- 将 JSON 和额外文字总结写入 `result.txt`。
- 将模型降级、输入异常等信息写入 `warnings` 或 `error` 字段。

### `src/main.py`

- 以项目根目录为基准定位 `resume.txt`、`jd.txt` 和 `result.txt`，不依赖启动时的工作目录。
- 自动读取输入并调用 Agent。
- 将 JSON 与文字总结打印到控制台。
- 输入缺失或为空时输出错误结果并以非零状态退出。

### 测试文件

- 保留并重写现有失效测试，新增确定性单元测试和端到端文件导出测试。
- 测试不依赖真实 Ollama 服务，通过注入假模型客户端验证成功和降级路径。

## JD 分类与技能词库

JD 分为三类：

1. 硬性门槛：学历、最低工作年限、专业。
2. 硬性技术技能：软件测试岗位要求的工具、语言、方法和流程。
3. 软素质：沟通、协作、责任心、学习能力、问题分析等。

首版软件测试词库至少包含以下规范技能：

- Python
- SQL
- pytest
- 测试用例
- 缺陷管理
- 接口测试
- 黑盒测试

匹配前统一大小写、全半角、空白及常见中英文写法。同义词映射至少包含：

- `pytest`、`PyTest`
- `测试用例`、`用例设计`、`test case`
- `缺陷管理`、`Bug 管理`、`缺陷跟踪`
- `接口测试`、`API 测试`
- `黑盒测试`、`功能测试`
- 按业务要求，将 `自动化测试`、`UI 自动化`、`接口自动化` 视为同一技能组

匹配输出统一使用规范技能名，避免同一能力重复计数。

## 评分模型

总分为 100 分，分项权重固定为：

- 硬性条件达标分：40 分。
- 技术关键词重合分：35 分。
- 项目经历契合分：25 分。

### 硬性条件达标分

仅对 JD 实际声明的学历、工作年限和专业要求计分，未声明的维度不参与分母；若 JD 没有任何硬性门槛，该项默认获得 40 分。

- 学历按中专/高中、大专、本科、硕士、博士层级比较。
- 工作年限比较简历中可验证的最大明确年限与 JD 最低年限。
- 专业按计算机、软件工程、软件测试及相关专业族匹配。
- 无法从简历确认满足条件时，按不满足处理，并说明“未提供可验证信息”。

硬性门槛不满足时执行总分封顶：

- 1 项不满足：总分最高 59 分。
- 2 项不满足：总分最高 39 分。
- 3 项及以上不满足：总分最高 19 分。

### 技术关键词重合分

以 JD 去重后的规范技能集合为分母，以简历中命中的规范技能集合为分子，按比例计算 35 分。若 JD 没有可识别技术技能，该项默认获得 35 分并写入提示，避免除零。

同义词组只能贡献一次分数。`matched_skills` 和 `missing_skills` 均按 JD 顺序去重输出。

### 项目经历契合分

项目分强调证据上下文，避免技能列表堆叠导致虚高：

- 60%：JD 技能是否实际出现在项目、职责、行动或成果上下文。
- 25%：是否体现测试活动，如用例设计、缺陷跟踪、接口验证、自动化执行和结果分析。
- 15%：是否在项目证据中体现 JD 软素质。

项目段落识别失败时，可使用职责或工作经历上下文，但仅技能清单中的词不能获得项目上下文部分的分数。

所有分项四舍五入为两位小数，总分先求和再应用硬性门槛封顶，并限制在 0 至 100 之间。

## STAR 改写与事实约束

STAR 改写使用完整 JD 条款、简历原始项目事实和缺失技能作为输入。

- Situation：只描述简历中已有的项目背景。
- Task：明确测试目标、职责或需要解决的问题。
- Action：自然融入简历中已有的测试技能和 JD 关键词。
- Result：优先使用简历原有成果；无量化数据时使用可验证的定性结果。
- `star_text`：将四部分合成为可直接放入简历的项目描述。

禁止行为：

- 不把缺失技能写成候选人已经掌握的技能。
- 不虚构项目、工具使用、工作年限、团队规模、性能数据或百分比成果。
- 不新增简历中不存在的数字。

缺失但重要的 JD 技能只进入 `supplemental_keywords`，并标注其用途或建议补充位置。模型失败时，规则模板按原始项目句子生成保守 STAR 结构，并在 `warnings` 中说明降级。

## 串行数据流

1. 读取并校验 `resume.txt` 和 `jd.txt`。
2. 规则抽取 JD 三类要求，必要时调用模型补充带原文证据的复杂条款。
3. 规范化技能与同义词，提取简历硬性条件和项目证据。
4. 计算三个分项、硬性封顶和总分。
5. 生成已匹配技能、缺失技能及不满足的硬性条件。
6. 调用模型生成 STAR 内容；异常时使用规则模板。
7. 根据缺失技能和 JD 条款生成增补关键词清单。
8. 组装 JSON 和中文总结，原子写入项目根目录 `result.txt`。
9. 将相同结果输出到控制台。

## 输出结构

JSON 主体字段固定为：

```json
{
  "total_score": 0,
  "scores": {
    "hard_requirements": 0,
    "technical_skills": 0,
    "project_experience": 0
  },
  "jd_analysis": {
    "hard_requirements": {
      "education": [],
      "work_years": [],
      "major": []
    },
    "technical_skills": [],
    "soft_qualities": []
  },
  "matched_skills": [],
  "missing_skills": [],
  "unmet_hard_requirements": [],
  "rewritten_project_experience": [
    {
      "situation": "",
      "task": "",
      "action": "",
      "result": "",
      "star_text": ""
    }
  ],
  "supplemental_keywords": [],
  "warnings": [],
  "summary": ""
}
```

`result.txt` 首先写入完整、可解析的格式化 JSON，随后空一行写入 `=== 文字总结 ===` 和中文总结。JSON 主体本身也保留 `summary` 字段，便于程序化消费。

发生输入错误时，JSON 至少包含：

```json
{
  "error": {
    "code": "INPUT_FILE_MISSING",
    "message": "缺少输入文件：resume.txt"
  },
  "warnings": [],
  "summary": "分析未执行。"
}
```

## 错误处理

- 输入文件缺失、不可读或为空：生成错误 JSON 与文字总结，不覆盖为伪造分析结果，并返回非零退出码。
- 模型连接失败、超时、空响应或非法 JSON：记录警告，继续确定性流程。
- 模型返回的字段缺失或类型错误：丢弃不可信字段，使用规则结果或模板默认值。
- 导出失败：控制台输出完整结果和明确错误，返回非零退出码。
- 日志不打印完整简历和 JD，避免泄露个人信息；只记录长度、步骤和异常类型。

## 测试与验收

自动化测试至少覆盖：

1. 学历、年限、专业达标与不达标判断。
2. 1、2、3 个硬性条件不满足时的 59、39、19 分封顶。
3. 软件测试技能词库、大小写归一化和同义词去重。
4. 技能在项目上下文中出现时的项目分高于仅在技能清单出现时。
5. 无技术要求或无硬性要求时的稳定分母处理。
6. 模型成功、连接异常和非法 JSON 三条路径。
7. STAR 输出字段完整，且不新增原简历中不存在的数字。
8. 串行调用顺序、JSON 必需字段和文字总结。
9. 从非项目工作目录启动时仍能读取项目根目录输入文件。
10. `result.txt` 生成位置、UTF-8 编码及内容格式。

验收命令优先使用标准库 `unittest` 运行全量测试，测试写法保持与 `pytest` 兼容；若项目环境已安装 `pytest`，再补充运行 `pytest`。随后用示例 `resume.txt`、`jd.txt` 执行 `src/main.py`，检查退出码、控制台输出与 `result.txt`。

## 非目标

- 不扩展为 Web 服务或恢复 Gradio 输入界面。
- 不引入外部数据库或在线技能词库。
- 不自动修改原始 `resume.txt`。
- 不为缺失事实生成虚构项目经历。
